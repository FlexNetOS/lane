Built in Rust  ·  tokio + hyper + rustls

Give localhost a
trusted local and public URL

lane maps custom domains to your dev-server ports with real HTTPS, path routing, and WebSocket passthrough for HMR — then shares any local port on a public URL with a single command.

myapp.test → localhost:3000
myapp.test/api → localhost:8080
dashboard.test → localhost:5173
app.loc → localhost:4000

Install with one command

$curl -sL https://lane.sh/install.sh | sh

macOS & Linux. Or build from source with Rust 1.82+.

Quick start

One command, a trusted HTTPS domain

The first run provisions a local CA, adds it to your OS trust store, and sets up port forwarding. After that, lane start is all it takes.

~/projects/web
$ lane start web --port 3000
✓ https://web.test → localhost:3000
$ lane start myapp --port 3000 --route /api=8080
✓ https://myapp.test → localhost:3000
https://myapp.test/api → localhost:8080
$ lane share --port 3000 # public tunnel
✓ https://cheeky-panda.lane.show → localhost:3000

Under the hood

How it works

A single binary that handles certificates, trust, port forwarding, and proxying — no per-project nginx, no hand-edited certs, no sudo on every request.

1

Local CA & OS trust

An RSA root CA, added to your system trust store, signs short-lived ECDSA leaf certs per domain — so https://*.test loads with no warnings.

2

Port forwarding

iptables on Linux and pf on macOS redirect 80→10080 and 443→10443, so nothing holds a privileged port.

3

hyper + rustls proxy

A detached daemon serves TLS on :10443 (HTTP/1.1 & HTTP/2), selects per-domain certs by SNI, and reverse-proxies to your local port.

4

WebSocket passthrough

Upgrade requests are bridged end to end, so Vite, Next.js, and other dev servers keep their HMR sockets alive over HTTPS.

Highlights

Everything you need, in one binary

Clean HTTPS domains for local dev, plus public sharing when you need to show someone.

Clean HTTPS domains

Trusted certificates, a green lock, and no browser warnings on every .test or custom-TLD domain you map.

Path routing

Send /api or /ws to a different upstream port on the same domain with --route /api=8080.

.lane.yaml projects

Declare every service for a project once, then bring them all up with lane up and tear them down with lane down.

Public tunnels

Expose a local port on a *.lane.show URL with custom subdomains, password protection, and a TTL for auto-expiry.

doctor diagnostics

Pass / warn / fail checks for the CA, OS trust, port forwarding, hosts entries, daemon, and leaf-cert expiry.

Zero-config first run

The CA, OS trust, and port forwarding are all set up automatically the first time you start a domain.

Reference

Commands

Everything is one lane command away.

lane start <name> Map a domain to a port
# Map your app to https://myapp.test $ lane start myapp --port 3000 # Use any TLD you like $ lane start app.loc --port 4000 # Route a path to a different upstream $ lane start myapp --port 3000 --route /api=8080 --route /ws=9000 # Wait for the upstream to come up first $ lane start dashboard --port 5173 --wait --timeout 30s
lane up  /  down Run a whole .lane.yaml project
# Start every service in the discovered .lane.yaml $ lane up # Point at a specific config $ lane up --config /path/to/.lane.yaml # Stop all project services $ lane down
lane share Public URL for a local port
# Random subdomain on lane.show $ lane share --port 3000 # Pick your own subdomain $ lane share --port 3000 --subdomain demo # Password-protect and auto-expire after 30m $ lane share --port 8080 --password secret --ttl 30m # Bring your own custom domain $ lane share --port 3000 --domain myapp.example.com
lane list  /  logs Inspect what's running
# Domains and tunnels with health status (alias: ls) $ lane list $ lane list --json # Tail access logs, follow one domain $ lane logs $ lane logs --follow myapp # Clear the log file $ lane logs --flush
lane domain Manage custom tunnel domains
# Requires: lane login $ lane login # Add, list, verify, and remove custom domains $ lane domain add myapp.example.com $ lane domain list $ lane domain verify myapp.example.com $ lane domain remove myapp.example.com
lane stop  /  doctor Tear down & diagnose
# Stop one domain, or stop everything $ lane stop myapp $ lane stop # Run diagnostic checks $ lane doctor # Update to the latest release (alias: update) $ lane upgrade # Remove the CA, certs, hosts entries, and config $ lane uninstall
